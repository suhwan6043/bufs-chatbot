# 캠챗 운영 매뉴얼 — 학사지원팀용

> **대상**: 학사지원팀 담당자 (비개발자)
> **목적**: 캠챗 일상 운영과 학생 피드백 대응을 학사지원팀에서 자력으로 처리
> **버전**: 1.0 — 2026-04-22 (파이널 미팅 4/27 직전 초판)
> **개발자 상세 매뉴얼**: `docs/OPS_내부팀.md`

## 1. 한눈에 보기 (30초 요약)

**캠챗**은 부산외대 학사 질문에 자동으로 답변하는 AI 챗봇입니다. 내부에 학사안내 PDF·공지사항·FAQ를 가지고 있고, 학생이 질문하면 그 중에서 답을 찾아 정리해 보여줍니다.

**학사지원팀의 역할**은 잘못된 답변을 받은 학생이 있거나, 답변 가능한 질문인데 시스템이 "모른다"고 한 경우 **관리자 화면에서 직접 보정**하는 것입니다. PDF·코드·시스템 자체를 바꾸는 일은 개발팀 몫입니다.

**핵심 원칙**

- 🟢 **내가 할 수 있음**: FAQ 추가·수정·삭제 / 학과 연락처 수정 / 크롤러 수동 실행 / 대화 로그 조회
- 🔴 **개발팀 호출**: 서버 재시작 / PDF 교체 / 코드·환경변수 변경 / 데이터베이스 직접 수정

## 2. 접속·로그인

| 항목 | 값 | 비고 |
|---|---|---|
| 챗봇 주소 | `https://maruvis.co.kr` (외부) · `http://[내부서버주소]` (내부망) | Cloudflare Tunnel 경유 HTTPS |
| 관리자 주소 | 챗봇 주소 + `/admin` | 예: `https://maruvis.co.kr/admin` |
| 관리자 비밀번호 | 대면 또는 암호관리자로 전달 | 메신저·메일 전달 금지 |
| 세션 타임아웃 | 30분 비활동 시 자동 로그아웃 | 재로그인 필요 |
| 로그인 실패 한도 | 5회 연속 실패 → 15분 잠금 | 기다리거나 수환에게 해제 요청 |

**비밀번호 분실 시**: 수환에게 요청 (대면·전화 권장, 메신저 지양)

## 3. 학사지원팀이 직접 할 수 있는 일 (🟢)

### 3.1 FAQ 관리 — 가장 자주 사용

**경로**: `/admin` 로그인 → `FAQ` 메뉴

**추가**
1. "새 FAQ" 클릭
2. 질문·답변 입력 (답변은 구체적으로, 학번·학과 조건이 있으면 명시)
3. 필요 시 관련 학과 선택
4. 저장 → **즉시** 학생 질문에 반영

**수정**
1. FAQ 목록에서 해당 항목 선택
2. 답변 수정 → 저장
3. 시스템이 자동으로 재인덱싱 (별도 조치 불필요)
4. 해당 FAQ를 과거에 질문했던 학생 세션에 "답변이 업데이트되었습니다" 알림이 자동 전송됩니다

**삭제**
1. 목록에서 선택 → 삭제
2. 즉시 검색에서 제외

**좋은 FAQ 작성 팁**
- 질문은 **학생이 실제로 쓰는 말**로 (공문체 지양)
- 답변은 **한 문단 + 근거 출처** 형식
- 학번·학과별로 답이 다르면 조건을 분리해서 별도 FAQ로 등록

### 3.2 학생 피드백 검토·답변 보정

**경로**: `/admin` → 피드백·미답변 관리

시스템이 자동으로 아래 조건을 **미답변 후보**로 분류합니다:
- 별점 2점 이하를 받은 질의 (`ADMIN_FAQ_RATING_THRESHOLD=2`)
- 답변에 "관련 정보를 찾을 수 없습니다" 문구가 포함된 질의
- 최근 7일 동안 수집 (`ADMIN_FAQ_SCAN_DAYS=7`)

비슷한 질문은 자동 그룹핑되어 중복 처리를 방지합니다 (자카드 유사도 0.6 이상).

**처리 절차**
1. 목록에서 질의 그룹 선택
2. 답을 알고 있으면 → "FAQ로 추가" 클릭 → 답변 작성 → 저장
3. 기존 FAQ와 매우 유사한 경우(0.75 이상) 경고가 뜸 → 기존 FAQ를 수정하는 쪽 권장
4. 답을 모르면 학과·부서에 문의 → 답 확정 후 FAQ 등록

### 3.3 크롤러 수동 실행

**경로**: `/admin` → 크롤러 관리

자동 주기(30분)를 기다리지 않고 **방금 올라온 공지**를 즉시 반영할 때 사용합니다.

1. "지금 실행" 클릭
2. 1~3분 후 크롤링 로그에서 새 공지 감지 확인
3. 10분 후 챗봇에 해당 공지 제목으로 질문해 반영 확인

### 3.4 학과 연락처 수정

**경로**: `/admin` → 학과 연락처 관리

학과명·전화·이메일·홈페이지 URL 수정 → 저장 → 즉시 답변에 반영.

### 3.5 대화 로그 조회

**경로**: `/admin` → 로그 (또는 `/logs` 페이지)

- 기간 선택 → CSV 다운로드
- 용도: 주간 질의 통계, 부실 답변 패턴 파악, FAQ 보강 우선순위 결정

## 4. 학사지원팀이 절대 하면 안 되는 일 (🔴 개발팀 호출)

| 작업 | 이유 |
|---|---|
| 서버(Streamlit · FastAPI · Ollama) 재시작 · 중단 | 타 사용자 세션 단절, 데이터 손상 가능 |
| `.env` 또는 `app/config.py` 수정 | 잘못된 설정은 서비스 전체 중단으로 이어짐 |
| `data/pdfs/` PDF 직접 교체·삭제 | PDF만 바꾸고 인제스트 누락 시 데이터 유실 |
| `data/chromadb/` 또는 `data/graphs/` 직접 수정 | 벡터DB·그래프 파손. 복구에 반나절 이상 |
| 코드 파일(`.py`) 수정 | 회귀 평가(164문항) 필요. 품질 하락 위험 |
| Git 커밋·푸시 | 프로젝트 원칙(CLAUDE.md)상 개발자 검증 후에만 |
| 원격 Ollama 서버(192.168.0.4) 조작 | 내부 인프라, 개발팀 외 접근 금지 |

## 5. 시나리오별 절차 5가지

### 시나리오 A — 학생이 "답이 이상해요"라고 신고
1. 학생의 **질문 원문과 답변 스크린샷** 확보 (시간·학번 포함)
2. `/admin` → 로그에서 해당 시점 세션 검색
3. 답변이 **명백히 틀림** → 올바른 답을 FAQ로 등록 (§3.1)
4. 답변이 **오래된 정보** (예: 이전 학기 기준) → 해당 PDF·공지가 최신인지 확인 → 아니면 수환에게 "재인제스트 요청"
5. 답변이 검색은 잘 했는데 **말투·문장만 이상** → 스크린샷만 개발팀에 공유 (프롬프트 튜닝 영역)

### 시나리오 B — 새 학기 공지가 홈페이지에 올라옴
1. 30분 이내면 자동 크롤러가 가져옴 → 우선 **기다리기**
2. 긴급(당일 처리 필요)하면 `/admin` → 크롤러 관리 → "지금 실행"
3. 10분 후 챗봇에 공지 제목으로 질문 → 반영 확인
4. 여전히 안 잡히면 공지 URL을 수환에게 전달

### 시나리오 C — 졸업요건이 바뀌었다
1. 공문·공지 근거 확보 (학과·학번·변경 내용을 명확히)
2. **FAQ로 해결 가능한 경우**: "25학번 영문과 필수학점" 같이 특정적인 질문 → 직접 FAQ 등록
3. **PDF 교체 필요**: 학사안내 PDF 전체가 바뀌는 경우 → 개발팀
4. **그래프 규정 변경**: 학점·학기·예외 조항 구조 자체가 바뀌면 개발팀

### 시나리오 D — FAQ 미답변이 쌓였다 (주간 루틴 권장)
1. 매주 지정 요일에 `/admin` → 피드백 관리에서 지난 주 미답변 후보 검토
2. 그룹별로: 답을 알면 FAQ 추가 / 학과 문의 필요하면 연락 후 확정
3. 답변 확정 FAQ는 자동으로 관련 학생에게 알림 전송됨

### 시나리오 E — 영어 질문에 엉뚱한 답
1. FAQ 답변 자체가 한국어만 있는 상태일 수 있음 → FAQ 수정 시 영어도 함께 입력 (선택)
2. 영어 용어가 시스템에 등록되지 않은 경우 → 개발팀에 "영어 용어 X 추가 요청"
3. 번역 자체가 오류 → 개발팀 (LLM·번역 튜닝 영역)

## 6. 문제 발생 시 1분 체크리스트

학생 신고 접수 시 아래 순서로 30초~1분 확인:

1. **나도 챗봇에 접속되나?** → ❌면 서버 다운 가능 → 수환 호출
2. **접속은 되는데 답변이 안 나오나?** → 원격 LLM 다운 가능 → 수환
3. **"관련 정보를 찾을 수 없습니다"가 뜨나?** → 지식 부족 → **FAQ 보정 가능** (§3.2)
4. **답변은 나오는데 내용이 틀림?** → FAQ·데이터 업데이트로 해결 가능 (시나리오 A)
5. **그 외** (성능 저하·깨진 레이아웃·로그인 문제) → 스크린샷 + 시간 기록 → 개발팀

## 8. 용어집

| 용어 | 설명 |
|---|---|
| RAG | Retrieval-Augmented Generation. 문서에서 찾아와 LLM이 답변하는 방식 |
| 크롤러 | 학교 홈페이지를 자동 순회하며 새 공지를 가져오는 프로그램 |
| 임베딩 | 문장을 숫자(벡터)로 바꿔 의미 유사도를 계산할 수 있게 하는 기술 |
| ChromaDB | 임베딩을 저장하고 유사 문장을 빠르게 찾는 데이터베이스 |
| 지식 그래프 | 학사 규정을 "노드(개념) + 엣지(관계)" 네트워크로 저장한 구조 |
| FAQ | 자주 묻는 질문의 정답을 미리 등록한 것. 가장 빠르고 정확한 답변 경로 |
| 인제스트 | 새 PDF·공지를 챗봇이 검색 가능하도록 처리(청킹→임베딩→저장)하는 과정 |
| LLM | Large Language Model. 답변 생성을 담당하는 생성형 AI |
| Ollama | LLM을 로컬·LAN 서버에서 실행하는 런타임 |
| 고정공지 | 학사공지 게시판의 상단 고정 공지 (📌). 검색 우선순위 상위 |
