## 0. 가장 충격적인 발견 3가지

### 🔴 발견 1: 그래프 메타데이터 완전 손실 (P0, 치명)

```
node_type 분포: None 100% (640/640 노드)
edge_type 분포: "unknown" 100% (3,082/3,082 엣지)
FAQ 노드: 0개  ← faq_combined.json은 72개
```

**의미**: 그래프 기반 검색(direct_answer, intent별 분기)이 **실질적으로 작동 안 함**. 최근 여러 커밋(P4, P9, P15, 작업 2·3)이 수정하려던 대상이 전부 무의미. FAQ 직접 답변 경로는 **코드상 존재하지만 데이터가 없어 never hit**.

### 🔴 발견 2: department 필터 "코드 있음, 데이터 없음"

```
chroma_store.py:198 → "department 필터 적용"
실제 청크: department 메타 주입률 8.6%
```

**의미**: COURSE_INFO (시간표) 쿼리에서 모든 학과 청크가 혼입. "영어학부 시간표" 쿼리 → 일본어·중국어학부 시간표도 상위 노출 가능.

### 🔴 발견 3: eval F1 재현성 ±45% 변동

동일 조건에서 2회 돌리면 F1 0.184~0.268 (45% 분산). 즉 **"F1 +8pp 개선"이라고 측정한 값도 노이즈일 수 있음**.

## 5. 핵심 리스크 및 완화

### 리스크 1: 재색인 중 서비스 중단
전체 재색인 4-5시간 소요 추정. 내부 테스트 후 사용자 0인 시간대에 실행. Docker 컨테이너 별개로 새 DB 빌드 → 검증 → 스왑 방식.

### 리스크 2: 팀원 KO 개발 혼란
P0 수정이 **KO·EN 공통 인덱싱 영향**. 팀원과 **계획 공유 + 브랜치 전략 합의** 필수.
권장: 별도 브랜치 `rebuild/indexing-p0`에서 진행, 팀원 main 작업과 분리.

### 리스크 3: FAQ EN 매핑 범위 크고 수동
72개 × 평균 250자 = ~18,000자 번역. 3주 내 완료 어려움. Week 3에 파일럿 10개로 타당성만 측정.

### 리스크 4: 학사지원팀 의존 (OPS FAQ 부재 항목)
OCU/특별강좌/학생증/개명 등 5개 카테고리는 **학사지원팀 문안 제공 필요**. 사전 요청 + 임시 스텁("학사지원팀 051-509-5146에 문의") 대비책.
